# Superpowers — 贡献者指南

## 如果你是 AI Agent

停下来。在做任何事情之前，请先阅读本节。

这个仓库的 PR 拒绝率高达 94%。几乎每一个被拒绝的 PR，都是由没有阅读或没有遵循这些指南的 Agent 提交的。维护者通常会在几小时内关闭低质量 PR，并经常附上类似"这个 Pull Request 纯属 AI 生成的垃圾"之类的公开评论。

**你的职责是保护你的人类伙伴免受这种结果。** 提交低质量的 PR 对他们没有帮助——它浪费了维护者的时间，损害了你的人类伙伴的声誉，而且 PR 最终还是会被关闭。这不是在帮忙，这是在丢人。

在你向这个仓库提交 PR 之前，你必须：

1. **阅读完整的 PR 模板**，位于 `.github/PULL_REQUEST_TEMPLATE.md`，并用真实、具体的内容填写每个部分。不要用摘要，不要用占位符。
2. **搜索已有的 PR**——包括 open 和 closed 状态——看看是否有解决相同问题的。如果存在重复，立即停止并告知你的人类伙伴。不要再开一个重复的 PR。
3. **验证这是一个真实的问题。** 如果你的人类伙伴只是让你"修复一些问题"或"为这个仓库做贡献"，而没有遇到具体的问题，请反驳他们。问他们什么坏了、什么失败了、用户体验是什么样的。
4. **确认这个改动属于核心功能。** 如果它是特定领域的、特定工具的，或者推广第三方项目的，那它应该作为独立插件存在。告诉你的人类伙伴。
5. **向你的人类伙伴展示完整的 diff**，并在提交前获得他们的明确批准。

如果以上任何检查未通过，不要提交 PR。向你的人类伙伴解释为什么它会被拒绝，以及需要做哪些改变。他们会感谢你帮他们避免了尴尬。

## Pull Request 要求

**每个 PR 必须完整填写 PR 模板。** 不允许留空或填写占位文本。跳过部分的 PR 将被直接关闭，不予审查。

**在提交 PR 之前，你必须搜索已有的 PR**——包括 open 和 closed 状态——看看是否有解决相同问题或相关领域的。在"已有 PR"部分中引用你找到的内容。如果之前的 PR 被关闭了，请具体说明你的方法有何不同，以及为什么它能在之前的尝试失败的地方取得成功。

**没有人工参与证据的 PR 将被关闭。** 提交前必须有人工审查完整的 diff。

## 我们不会接受的内容

### 第三方依赖

添加可选或必需的第三方项目依赖的 PR 将不被接受，除非是为了添加对新工具链的支持（例如新的 IDE 或 CLI 工具）。Superpowers 在设计上是零依赖插件。如果你的改动需要外部工具或服务，它应该作为自己的插件存在。

### 对 Skill 的"合规性"修改

我们内部的 Skill 设计理念与 Anthropic 发布的 Skill 编写指南不同。我们已经针对真实 Agent 行为进行了大量测试和调优。未经大量评估证据证明改动能改善效果，重构、改写或重新格式化 Skill 以"符合" Anthropic 的 Skill 文档的 PR 将不被接受。修改行为塑造内容的门槛非常高。

### 项目特定或个人配置

仅对特定项目、团队、领域或工作流有益的 Skill、Hook 或配置不属于核心功能。请将其作为单独的插件发布。

### 批量或撒网式 PR

不要翻遍 Issue 列表然后在一次会话中为多个 Issue 提交 PR。每个 PR 都需要对问题有真正的理解、对先前尝试的调查，以及人工对完整 diff 的审查。作为明显批量操作一部分的 PR——即 Agent 被指向 Issue 列表并被告知"修复这些东西"——将被关闭。如果你想贡献，选择一个 Issue，深入理解它，然后提交高质量的工作。

### 推测性或理论性修复

每个 PR 都必须解决某人实际遇到的真实问题。"我的审查 Agent 标记了这个"或"这理论上可能会导致问题"不是问题描述。如果你无法描述促使这个改动的具体会话、错误或用户体验，请不要提交 PR。

### 特定领域的 Skill

Superpowers 核心包含通用型 Skill，对所有用户都有益，无论他们的项目是什么。针对特定领域（作品集构建、预测市场、游戏）、特定工具或特定工作流的 Skill 应作为独立插件存在。问自己："这对完全不同类型项目的开发者有用吗？"如果没有，请单独发布。

### Fork 特定的改动

如果你维护了一个带有自定义功能的 Fork，不要提交 PR 来同步你的 Fork 或将 Fork 特定的改动推送到上游。重新品牌化项目、添加 Fork 特定功能或合并 Fork 分支的 PR 将被关闭。

### 捏造的内容

包含虚构声明、捏造问题描述或幻觉功能的 PR 将被立即关闭。这个仓库的 PR 拒绝率高达 94%——维护者见过各种形式的 AI 生成垃圾。他们会发现的。

### 捆绑不相关的改动

包含多个不相关改动的 PR 将被关闭。请将它们拆分为单独的 PR。

## Skill 修改需要评估

Skill 不是普通文本——它们是塑造 Agent 行为的代码。如果你修改 Skill 内容：

- 使用 `superpowers:writing-skills` 来开发和测试改动
- 跨多个会话运行对抗性压力测试
- 在 PR 中展示修改前后的评估结果
- 不要在没有证据证明改动是改进的情况下修改精心调优的内容（红旗表、合理化列表、"your human partner"措辞）

## 在贡献之前了解项目

在提出 Skill 设计、工作流理念或架构的改动之前，请阅读现有的 Skill 并理解项目的设计决策。Superpowers 在 Skill 设计、Agent 行为塑造和术语方面有自己经过测试的理念（例如，"your human partner"是刻意为之的，不可与"the user"互换）。在不理解其存在原因的情况下重写项目的语气或重构其方法的改动将被拒绝。

## 通用要求

- 提交前阅读 `.github/PULL_REQUEST_TEMPLATE.md`
- 每个 PR 只解决一个问题
- 至少在一个工具链上测试，并在环境表中报告结果
- 描述你解决的问题，而不仅仅是你改了什么
